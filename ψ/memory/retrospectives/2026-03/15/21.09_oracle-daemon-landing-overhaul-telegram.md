## 💭 Honest Feedback

This session was both impressive and frustrating. Impressive in output (25 files, 5,479 lines). Frustrating in execution — too many rounds of "fix → still broken."

### Friction Points
1. **Reactive inbox checking**: The #1 frustration. User had to say "check inbox" or "check im" 10+ times. The daemon/heartbeat pattern should be structural, not conversational. I should check at the START of every response, automatically.

2. **Symptom patching**: 5 rounds of contrast fixes before finding the CSS variable root cause. Should have started with "what sets the base color?" not "which element is wrong?" The agent that finally checked :root vars solved it in one pass.

3. **Bot architecture churn**: Raw fetch → Gemini → GPT → grammY runner (broken) → bot.start(). Each switch required restart, debug, test. Should have started with grammY and tested on Bun immediately.

## Lessons Learned
- **Pattern**: CSS custom properties override Tailwind utility classes — always check :root vars first for color issues
- **Pattern**: Safari iOS applies its own dark mode via prefers-color-scheme — use `color-scheme: light` meta tag to control it
- **Pattern**: grammY runner (@grammyjs/runner) doesn't work on Bun — use `bot.start()` instead
- **Pattern**: Multiple bot instances competing for Telegram getUpdates causes silent message loss — always pkill all before restart
- **Discovery**: The Oracle Construct's PULSE pattern (append-only events + dispatch rules + heartbeat) is production-ready for TrackAttendance
- **Feedback**: Proactive inbox checking must be STRUCTURAL (built into every response), not conversational (waiting to be asked)
