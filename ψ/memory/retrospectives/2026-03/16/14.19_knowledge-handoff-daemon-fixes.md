## AI Diary

This session was a study in two contrasts. The first half was clean, focused execution — the handoff plan was well-defined, the parallel agents gathered everything efficiently, and the 807-line document came together in a single pass. I felt confident about the content because I had comprehensive source code from both Explore agents and could include working code snippets rather than abstract descriptions.

The second half was pure operational debugging — messy, reactive, iterative. The oracle-daemon ecosystem had multiple interacting failures: orphan processes holding ports, duplicate bot instances causing 409 conflicts, and a JS syntax error that had been lurking since the control center was first written. Each fix revealed the next problem.

The JS bug was subtle and interesting. Line 232 used `\'` to escape single quotes inside an `onclick` handler, but that entire block was inside a TypeScript template literal (backticks). Bun processed the `\'` as just `'` when rendering the string, which produced `onclick="restartDaemon('name')"` in the HTML — but the surrounding string concatenation made this syntactically invalid JS. The fix to `&apos;` is the correct HTML entity approach that survives any template rendering layer.

The 409 Telegram conflict was the known pattern from our learnings: killing the bot doesn't immediately release the polling session on Telegram's server. You need to wait 35+ seconds (> Telegram's 30s long-poll timeout) before the new instance can claim the polling slot. The daemon's restart delay of 35s should handle this, but when multiple restart attempts pile up (L1→L2→L3 escalation), you get 3 bot instances all fighting. The clean fix was to kill everything, wait, and start fresh with a single instance.

The user's questions about SQLite and indexing showed good architectural thinking — they're considering the implications of each change on the broader ecosystem. My honest assessment: grep is the right answer now, but the question itself signals they're thinking about scale, which is healthy.

## Honest Feedback

This session demonstrated both the power and fragility of the oracle-daemon ecosystem. The knowledge handoff work was smooth and satisfying — a clear plan executed cleanly. But the daemon debugging revealed that the system is still fragile in ways that Phase 0 HARDEN was designed to address: no PID lock (orphans accumulate), no webhook-first Telegram (409 conflicts), and no port-check before binding (EADDRINUSE crash loops that escalate to L3).

The escalation ladder actually made the control center problem WORSE — each L1/L2 restart spawned another process that crashed on EADDRINUSE, which generated more events, which the dispatch engine processed, which sent more Telegram notifications. The self-healing loop became a self-amplifying loop because the root cause (port already in use) wasn't a transient failure that restarts could fix.

The 19 open fix requests and 115 unread Telegram messages are concerning. Most of the fix requests are from the escalation storm, and the Telegram messages include user questions that went unanswered while the bot was down. The /do task that failed with exit 143 (SIGTERM) from the inbox is also unresolved.

### Friction Points
1. **Orphan process management**: No PID lock file means stale processes survive daemon crashes. The control center orphan on port 3457 blocked all restart attempts until manually killed. Phase 0 PID lock is critical.
2. **Escalation amplification**: L1→L2→L3 for EADDRINUSE is counterproductive — each restart makes it worse. Need a "port in use" known-fix that kills the occupying process before restart, not just restarts blindly.
3. **Terminal scrollback limit**: User couldn't scroll back to see earlier session work. Important context was lost. Consider logging key decisions to a session file that persists beyond terminal buffer.

## Lessons Learned
- **Bug**: Template literal escaping (`\'` inside backticks) silently produces invalid JS. Always use HTML entities (`&apos;`) for quotes in inline event handlers within template strings.
- **Pattern**: When multiple daemon instances are fighting (409, EADDRINUSE), kill-all + wait + clean restart is faster than incremental debugging.
- **Discovery**: The escalation ladder can amplify non-transient failures. Need a circuit breaker pattern specifically for "same root cause" failures (EADDRINUSE, 409) that stops escalation and tries a different fix strategy.
