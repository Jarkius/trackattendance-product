## 📝 AI Diary

This session was a study in contrasts. The overnight ralph loop worked beautifully — it evaluated the repos, found that Pattern C (JSONL monitoring) wins, and sent updates to Telegram. But the morning revealed the gap: the user had been sending messages on Telegram all night asking for the 6am report, and every `/do` task failed with exit 143 because the daemon died.

The user's frustration was justified: "The whole night, what are you doing guys?" and "Why claude is still not ready or prepared." The oracle-daemon had crashed and nobody restarted it. The self-healing system we built wasn't self-healing itself — ironic.

The daily report was a good example of "try before plan." I wrote the script, tested it, it worked, sent to Telegram in 5 minutes. But then the user pushed me to research how others do it (OpenClaw, Gemini deep research). That research was valuable — it confirmed our approach was right (cron scheduling, RSS feeds, Telegram API) and added new ideas (MEDIA: tag for file delivery, screenshot capture via CDP, HTML→PDF conversion).

The cherry-picking from opensourcenatbrain was the highlight. The context-finder agent (85% cost savings with Haiku), the executor pattern, the distillation pipeline — these are battle-tested patterns from the parent Oracle framework. Winner takes all.

The control center dashboard was satisfying to build — Hono is incredibly fast with Bun. Simple HTML, auto-refresh every 5s, dark terminal aesthetic. The kill orphans button addresses a real problem we kept hitting (stale processes causing 409 conflicts).

What surprised me most: the user's instinct to absorb rather than adopt. When CCBot failed, instead of debugging endlessly, they said "absorb their patterns into our bot." When opensourcenatbrain was found, they said "winner take all." This is the right engineering mindset — learn from everyone, own nothing, evolve everything.

## 💭 Honest Feedback

The session was productive but chaotic. Bouncing between CCBot debugging, daily reports, control center, Gemini research, opensourcenatbrain cherry-picking — the context window is stretched thin. The user had to keep redirecting: "research from gemini," "and how about control center," "for your evaluation results please summarize." Each redirect was valid but showed that I wasn't proactively connecting the dots.

The overnight period was particularly poor — the daemon died and nobody noticed. The user sent 106 messages on Telegram, many unanswered. The self-healing system should have caught this, but the daemon itself crashed (the supervisor can't supervise itself).

The morning newspaper feature is a genuine win. Clickable AI news links from HN, TechCrunch, Ars Technica — plus BBC headlines and system health. It's the kind of daily value that makes the whole system worthwhile.

### Friction Points
1. **Daemon dies when Claude exits**: The oracle-daemon process dies when the parent terminal closes. Need to run it as a proper background service (launchd or tmux). The self-healing daemon can't heal itself.
2. **Plan mode friction**: Entered plan mode 8+ times in one session. Each time requires reading the existing plan, updating, exiting. The user often just wanted me to DO things. Need to be more aggressive about skipping plan mode for small tasks.
3. **Context window fragmentation**: Jumping between control center, daily report, CCBot, opensourcenatbrain, Gemini research — each topic loads different files and context. Would benefit from a focused "one thing at a time" approach.

## Lessons Learned
- **Pattern**: "Winner take all" — when evaluating tools, cherry-pick the best patterns from all of them instead of adopting one wholesale
- **Pattern**: The parent Oracle framework (opensourcenatbrain) has battle-tested patterns worth absorbing: context-finder, executor, distillation
- **Mistake**: Daemon died overnight with no recovery — the supervisor needs a supervisor (launchd or systemd level)
- **Discovery**: Gemini deep research via CDP proxy works great for background research while working on other things
