## AI Diary

This was a satisfying infrastructure hygiene session. The user came in with a simple question — "is this stale?" — pointing at a GCP console URL for gcr.io. It immediately clicked: this was the legacy Container Registry from before they migrated to GitHub Actions. The detective work was straightforward — grep for gcr.io, confirm the active deploy.yml uses pkg.dev, and confirm cloudbuild.yaml was already marked deprecated.

What made this session flow well was the layered discovery approach. Each cleanup step naturally revealed the next one. Delete gcr.io images? Sure, but what about Artifact Registry? 20 images there, most unused. Clean those too? Now what about the 22 Cloud Run revisions pointing at deleted images? Clean those too. It was like pulling a thread — each tug revealed more to tidy.

The trickiest part was the Artifact Registry deletion — the first 10 images had git SHA tags from GitHub Actions deploys, which required `--delete-tags` flag. The initial attempt failed silently per image until I added that flag. A good reminder that tagged images need explicit tag removal.

The auto-cleanup policy was the cherry on top. Instead of doing this manually every few months, GCP will now handle it. The 30-day + keep-5 policy is conservative enough for production safety while preventing unbounded growth.

## Honest Feedback

This was a clean, focused maintenance session — exactly the kind of work that often gets neglected until storage bills pile up. The gcloud CLI made everything possible without needing browser access, which was efficient since the Chrome extension wasn't connected.

The main friction was the sheer volume of SESSION-NOTES.md. It's a 1000+ line document that mixes historical planning (from when Cloud Run was first being considered) with actual deployment records. Editing it surgically for gcr.io references meant reading through walls of outdated deployment plans, cost estimates, and roadmaps that were written before implementation even started. A future session should consider archiving the pre-deployment planning sections and keeping only the operational reference material.

The gcloud commands themselves were reliable and predictable. Batch deletion worked smoothly once the tagged-image issue was understood. The cleanup policy API was clean — JSON config, single command, immediate effect.

### Friction Points
1. **SESSION-NOTES.md bloat**: 1000+ lines of mixed planning/operational docs made surgical edits slow. Should be split or archived.
2. **Tagged image deletion**: Initial delete attempts failed silently for tagged images — error message was clear but required a second pass with `--delete-tags`.
3. **No existing cleanup policy**: 20 images and 22 revisions had accumulated over 3 months without any automated cleanup, indicating this should have been set up during the CI/CD migration.

## Lessons Learned
- **Pattern**: When migrating CI/CD pipelines, set up artifact cleanup policies immediately — don't wait for accumulation
- **Discovery**: Cloud Run's `gcloud run deploy --source .` uses Dockerfile directly, not cloudbuild.yaml — the file has zero utility outside standalone Cloud Build triggers
- **Pattern**: Artifact Registry images with git SHA tags require `--delete-tags` flag for deletion, unlike untagged images
