## 💭 Honest Feedback

This was a productive session with clear user-driven priorities. Each feature was small, well-scoped, and testable. The conversation flow was natural — user reports issue, we investigate, fix, test, move on.

The duplicate detection improvement was the most impactful change. It's a subtle edge case (same person, different input methods) but in a real event with multiple operators, it would definitely happen. The fix was clean — just an additional query by Legacy ID after the badge_id check.

The roster blocking decision was good product thinking from the user. Rather than trying to handle bad data gracefully, just refuse it and give the admin clear instructions to fix it. This prevents cascading issues during events.

### Friction Points
1. **Test artifacts left behind**: Created `attendance.db` during quick testing that confused the user. Should use temp directories or clean up immediately after test scripts.
2. **Mtime cache blocks testing**: Had to manually delete roster_meta rows to force reimport. Could add a `--force-reimport` CLI flag for testing.
3. **Multiple deploy cycles**: Dashboard changes require Cloud Run deploy (~5 min each). Batching changes before deploying would be more efficient.

## Lessons Learned
- **Pattern**: Duplicate detection should check by resolved entity ID, not raw input — different input methods can refer to the same entity
- **Pattern**: Block bad data at import time with clear error messages, not handle it gracefully downstream — prevents cascading issues
- **Discovery**: SQLite stores `legacy_id` as a separate column in scans table — enables employee-level queries independent of input method
