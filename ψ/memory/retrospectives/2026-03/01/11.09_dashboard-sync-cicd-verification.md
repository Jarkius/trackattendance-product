### Friction Points
1. **Workspace path confusion**: Two `projects-office/trackattendance/` directories exist (one empty wrapper, one with code). This caused the venv to break and confused the user's terminal navigation. Suggestion: remove or symlink the old empty wrapper.
2. **No requirements-test.txt**: The frontend requirements.txt includes PyQt6, pythonnet, and other platform-specific packages that won't install on Linux CI runners. Had to cherry-pick pip packages for the CI workflow instead of `pip install -r requirements.txt`. Suggestion: create a `requirements-test.txt` with just the test dependencies.
3. **API has no test suite yet**: The type check catches compilation errors but not logic bugs. The API endpoint behavior (batch dedup, BU aggregation, roster sync) is only verified by manual stress tests. Suggestion: add Vitest with basic endpoint tests.

## Lessons Learned
- **Pattern**: Venv path binding — Python venvs hardcode the creation path in pyvenv.cfg and bin/activate. Moving the project directory breaks the venv silently (activate runs but python resolves to system).
- **Pattern**: Dashboard verification loop — comparing local SQLite stats with cloud API stats after sync is a reliable end-to-end smoke test. More valuable than unit tests for integration confidence.
- **Discovery**: GitHub Actions auto-deploy makes static file iteration feel serverless — push HTML changes and see them live in ~2 minutes without any manual deploy step.
