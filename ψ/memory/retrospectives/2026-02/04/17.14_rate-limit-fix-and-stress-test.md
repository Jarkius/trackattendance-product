- Changed `app.register(rateLimit, ...)` to `await app.register(rateLimit, ...)`
- Changed `app.listen().then(...)` to `await app.listen()` inside bootstrap

### Architecture Decisions
- Used async bootstrap pattern rather than Fastify's `.after()` callback chain — cleaner and more readable
- Kept `skipOnError: true` so rate limiter failures don't block legitimate traffic

## AI Diary

This session was a satisfying debugging exercise. The rate limiter was "installed" but silently doing nothing — a particularly insidious class of bug where everything looks correct but the behavior is wrong. My first instinct was to check if `skipOnError: true` was masking failures, but the real issue was more fundamental: Fastify plugin registration is async, and without `await`, the routes were being added before the rate limit plugin had finished initializing.

What made this click was writing a minimal isolated test. The first test (without `await`) showed all requests passing. The second test (with `await`) immediately showed 429s. That's the kind of clarity you only get from reduction to minimal reproduction. The fix was straightforward once the cause was clear — wrap everything in an async function.

The stress test was gratifying. 1,000 scans across 10 concurrent kiosks, all saved in 1.2 seconds with sub-200ms average latency. Then the burst test confirmed the rate limiter blocks correctly: 62 out of 70 rapid requests got 429. The system behaves exactly as designed — normal kiosk traffic flows freely while abuse gets blocked.

One interesting observation: the rate limit window reset caught us when trying to clean up test data. The DELETE request got 429 because we'd just fired 70 requests. Had to wait 50 seconds for the window to reset. This is actually good — it proves the rate limiter works even for admin endpoints, providing defense-in-depth.

## Honest Feedback

This was a focused, efficient session. The debugging was quick because the right approach was used immediately — minimal isolated reproduction rather than guessing. The stress test gave confidence that the system handles real-world concurrent load.

The main friction was inheriting a bug from the previous session that should have been caught during initial implementation. When adding Fastify plugins, the async registration pattern should always be used — this is documented in the Fastify docs but easy to miss when writing top-level module code in TypeScript where `await` at module scope isn't available.

The Cloud Run deploy cycle (~3 minutes build + deploy) is reasonable but adds up. Having the local test confirm the fix before deploying saved a potential round-trip.

## Lessons Learned
- **Pattern**: Fastify plugin `await` — always use `await app.register()` or wrap in async bootstrap, especially for plugins that add hooks/decorators used by routes
- **Discovery**: Cloud Run rate limiting works per-instance with in-memory store, but with single instance it enforces correctly
- **Verification**: Isolated minimal reproduction is the fastest path to root cause
