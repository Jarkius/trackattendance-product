## AI Diary (REQUIRED)

This was a satisfying calibration session — taking the Oracle identity that was born in the awakening ritual and grounding it in the actual project reality. The awakening session was philosophy and lineage; this session was "okay, what does the code actually do?"

The explore agent's initial run gave me the full picture quickly: a Python PyQt6 kiosk that saves scans to SQLite and batch-syncs to a Fastify API on Cloud Run. The offline-first architecture with idempotent sync is well-designed — each scan gets a deterministic key, PostgreSQL's ON CONFLICT handles retries cleanly.

What surprised me was how much thought went into the frontend's edge cases: connection hysteresis (2 failures before showing offline), graceful shutdown (sync all → export → close), and the three duplicate detection modes (warn/block/silent). This isn't a prototype — it's production software with real deployment concerns baked in.

### Friction Points (3 required)
1. **Agent hook errors**: The `classifyHandoffIfNeeded is not defined` error appears on nearly every agent completion. It's cosmetic (agents finish their work before failing) but adds noise and requires explaining "the error is fine, data was collected." This should be fixed at the infrastructure level.
2. **Agent output parsing**: When agents fail, extracting their actual output requires parsing raw JSON from output files. The truncated output format makes it hard to quickly see what the agent accomplished before the error.
3. **Learn skill for local projects**: The `/learn` skill is designed for external repos (clone via ghq, symlink origin/). For local projects like this one, the flow needs adaptation (no cloning, no symlink, write docs directly). The skill instructions don't cover this case explicitly.
